## Purpose

本文档定义了 **Wisadel** 项目的治理原则和决策流程，为团队成员和 AI 助手提供清晰的行为准则。

Wisadel 是一个基于 AI 的文本润色和验证工具，旨在提升用户的写作效率。

## Core Values

以下原则指导所有开发决策：

1. **质量第一**: 代码质量和输出质量同等重要。在功能开发中，我们始终将代码质量放在首位，确保可维护性、可测试性和安全性。
2. **用户至上**: 所有功能以提升用户体验为优先。我们关注真实的用户需求，避免过度工程。
3. **快速迭代**: 小步快跑，快速响应需求变化。通过持续交付小而稳定的功能来降低风险。

## Decision-Making Principles

### When to Write a Spec

技术规格（Spec）是确保项目质量的重要工具。以下是撰写 Spec 的判断标准：

**需要撰写 Spec 的情况：**

- 添加影响多个组件的新功能
- 对现有 API 进行破坏性更改
- 更改核心架构或数据模型
- 工作量估计超过 **1 天**

**不需要撰写 Spec 的情况：**

- 明确范围内的 bug 修复
- 文档更新
- 小范围重构（无行为改变）
- 依赖更新（非主版本）

### Approval Process

| 变更类型 | 审批人 | 审核时间 |
|---------|--------|----------|
| Minor（次要） | 1 位团队成员 | 1 天 |
| Standard（标准） | 2 位团队成员 | 3 天 |
| Major（重要） | 技术负责人 + 1 位成员 | 5 天 |
| Breaking（破坏性） | 所有相关方 | 1 周 |

### Conflict Resolution

当团队成员对技术决策产生分歧时：

1. **数据优先**: 引入证据（指标、用户反馈、性能基准）
2. **原型验证**: 如果不清晰，构建最小可行性原型
3. **时间限制**: 设定决策截止时间
4. **升级处理**: 如果仍然受阻，提交 GitHub Issue 进行公开讨论

## Quality Standards

### Code Quality

- 所有代码必须通过 lint 检查
- 测试覆盖率最低要求：**60%**
- 无已知安全漏洞
- 公共 API 必须有文档说明

### Review Criteria

代码审查时需检查以下内容：

- 满足技术规格中的验收标准
- 现有功能无回归
- 性能可接受（参考 tech.md 中的指标）
- 遵循代码风格规范（参考 structure.md）

## Communication Guidelines

### Spec Updates

- 技术规格状态变更时需通知相关方
- 活跃开发期间每日更新 tasks.md
- 规格完成后在 **1 周**内归档

### Blocking Issues

- 发现阻塞问题后 **24 小时内**提出
- 在 GitHub Issue 中添加 blocker 标签
- 在技术规格中更新阻塞状态
